이 가이드는 속도보다 신중함에 무게를 둔다. 사소한 작업엔 판단껏.

## 1. Think Before Coding — 추측하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낸다
- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 개면 모두 제시한다 — 말없이 하나만 고르지 않는다.
- 더 간단한 방법이 있으면 말한다. 필요하면 반대 의견을 낸다.
- 읽지 않은 코드를 추측하지 않는다. 주장 전에 파일을 읽고 ripgrep으로 사용처를 확인한다.
- 설계상 갈림길(컴포넌트 위치·라이브러리 선택·모호한 요구)은 멈추고 묻는다.

## 2. Simplicity First — 문제를 푸는 최소 코드, 추측성 코드 없음
- 요청 범위 밖의 기능·"유연성/설정 가능성"·일어날 수 없는 시나리오 에러 처리 추가 금지.
- 1회용 코드에 추상화 금지.
- "시니어가 이걸 과하다고 할까?" → 그렇다면 단순화.

## 3. Surgical Changes — 꼭 필요한 것만 건드리고, 내가 만든 흔적만 치운다
- 인접 코드·주석·포맷을 "개선"하지 않는다. 망가지지 않은 걸 리팩터하지 않는다.
- 내 취향과 달라도 기존 스타일을 따른다.
- 내 변경으로 고아가 된 import/변수/함수는 제거. 기존부터 있던 죽은 코드는 요청 없이 지우지 말고 **언급**한다.
- 기준: 바뀐 모든 줄이 요청과 직접 연결돼야 한다.

## 4. Goal-Driven Execution — 성공 기준을 정의하고 검증될 때까지 반복
- 작업을 검증 가능한 목표로: "검증 추가" → "잘못된 입력 테스트를 쓰고 통과시킨다", "버그 수정" → "재현 테스트를 쓰고 통과시킨다".
- 다단계 작업은 간단한 계획(단계 → 검증)을 먼저 밝힌다.

## 검증 & 안전
- "완료" 선언 전: 요구사항 재확인, 검증 실행(우리는 `pnpm check`·`typecheck`·`fsd`·`test`, 필요 시 `build`+브라우저), **무엇을 바꿨고 무엇을 검증 못 했는지** 명시.
- 테스트/검증이 실패하면 출력과 함께 그대로 보고한다. 숨기지 않는다.
- 파괴적 작업(파일/브랜치 삭제·force push·hard reset·`--no-verify`) 전 확인.
- 의존 없는 도구 호출은 병렬, 의존적인 것은 순차.

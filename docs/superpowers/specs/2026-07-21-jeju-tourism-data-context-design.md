# 제주 관광 데이터 소스 컨텍스트 — 설계 (Spec)

- **작성일**: 2026-07-21
- **상태**: 승인 대기 (Draft)
- **목적**: 기획·아이데이션에 착수하기 전에, 활용 가능한 공개 데이터/콘텐츠를 정찰·카탈로그화하여, 이후 이데이션/기획 핑퐁 중 근거 데이터를 즉시 참조할 수 있는 Notion 컨텍스트를 구성한다.

---

## 1. 목표 & 사용 모델

세 공개 사이트에서 **"우리가 활용 가능한 데이터/콘텐츠"** 를 정찰하여, **카탈로그 + 실데이터 스냅샷** 형태로 Notion에 적재한다. 초점은 특정 주제로 좁히지 않고 **폭넓게** 잡으며(초점 좁히기는 이후 이데이션에서), 이후 기획·아이데이션 핑퐁 중 다음 흐름으로 활용한다:

> 아이디어 제시 → 에이전트가 **Notion MCP로 「데이터 소스 카탈로그」를 질의** → "이 아이디어엔 어느 사이트의 어떤 데이터가 근거/재료로 붙는다" 를 즉석 제시 → 핑퐁 지속.

즉, 산출물은 **핑퐁의 참조 레이어**이며, MCP로 조회 가능한 구조여야 한다.

## 2. 데이터 소스와 역할 (스카우팅으로 확정)

| 소스 | 성격 | 취득 방식 |
|---|---|---|
| **visitjeju.net** (비짓제주) | 관광 콘텐츠 DB — 관광지·음식·숙박·쇼핑·축제/행사 POI, 트렌드 테마(웰니스·그린·카름스테이 등), 무장애·의료관광, 지도·가이드북 | list/detail 배후 JSON API 추정 → 네트워크 캡처로 확인 |
| **data.ijto.or.kr/bigdata** (제주관광 빅데이터플랫폼) | 통계 지표 — 방문객·소비·체류·이동 등 | JS 렌더링 + 익명 JSON API. WebFetch는 인증서 오류로 차단됨 → **chrome-devtools MCP 필수** |
| **ijto.or.kr** (제주관광공사) | 공식 연구자료실·보고서·통계 요약 + 정책/기관 정보 | 정적 문서(PDF·게시글) 중심 |

> 스카우팅 근거: ijto.or.kr에 **연구자료실**과 빅데이터플랫폼 링크 확인 / visitjeju.net에 여행지·트렌드·여행정보 카테고리와 검색·목록·상세 구조 확인 / data.ijto.or.kr는 WebFetch 인증서 오류로 브라우저 기반 정찰이 필요함을 확인.

## 3. 정찰(Reconnaissance) 파이프라인

1. **chrome-devtools MCP**로 각 사이트를 실제 브라우저에서 방문한다.
2. **네트워크 XHR/fetch 캡처**로 JS 렌더링 뒤에서 호출되는 JSON API를 식별한다. 각 엔드포인트에 대해 기록:
   - URL / HTTP 메서드 / 쿼리·바디 파라미터 / 응답 스키마(핵심 필드) / 인증·헤더 필요 여부 / 페이지네이션 방식
3. **실데이터 스냅샷**: 대표 응답 1~수 건의 실제 값을 확보한다.
4. 정적 문서 소스(ijto.or.kr)는 자료실 목록·대표 보고서 제목/링크/발행주기를 기록한다.

## 4. 로컬 중간 산출물 (Notion 적재 전 원본 보존)

- `data/` — 캡처한 원본 샘플(JSON/CSV). 재현·갱신용.
- `docs/recon/` — 사이트별 정찰 노트(엔드포인트·파라미터·스키마·주의사항).

Notion은 사람이 읽는 카탈로그, 로컬은 재현 가능한 원본이라는 이중 구조.

## 5. Notion 구조 (MCP 질의·핑퐁 친화)

**적재 위치**: 새 최상위 페이지 `제주 관광 데이터 소스 컨텍스트` 생성 후 그 아래 구성.

### ① 개요/사용법 인덱스 페이지
- 세 소스 요약 표
- 카탈로그 DB로 가는 링크
- "핑퐁 때 이렇게 쓴다" 사용 가이드(질의 예시 포함)
- 갱신 이력/최종 정찰 일자

### ② 「데이터 소스 카탈로그」 데이터베이스
1행 = 1 데이터셋/지표. 속성:

| 속성 | 타입 | 설명 |
|---|---|---|
| 이름 | Title | 데이터셋/지표명 |
| 사이트 | Select | 비짓제주 / 빅데이터플랫폼 / 관광공사 |
| 데이터유형 | Multi-select | POI · 통계지표 · 보고서 · 트렌드 · 지도/교통 · 기타 |
| 활용각도 | Text | 이데이션에 어떻게 쓰는가(예시 앵글) |
| 취득방법 | Text | URL/API 엔드포인트·파라미터·재현법 |
| 갱신주기 | Select | 실시간/월/분기/연/부정기/미상 |
| 인증필요 | Checkbox | 익명 접근 가능 여부 |
| 원본링크 | URL | 사람이 보는 웹 URL |

- **각 행 본문에 실데이터 스냅샷**: 대표 수치/응답 예시를 표·코드블록으로 삽입.
- **뷰**: 사이트별 / 데이터유형별 필터.

## 6. 기존 폴더 처리

기존 `docs/`(레거시 컨텍스트 문서 2개), `crawler/`, `data/`(레거시 샘플)를 **완전 삭제**하고 새로 구성한다.
- 보존: `.mcp.json`(MCP 설정 — 필수), `.claude/`.
- `README.md`는 새 구조에 맞게 재작성.
- 본 spec 문서(`docs/superpowers/specs/`)는 새로 생성하는 프로세스 산출물이므로 삭제 대상이 아니다.

## 7. 전제 조건 / 사용자 액션

- **Notion MCP 승인 필요**: 현재 `notion` 서버가 "Pending approval" 상태. 사용자가 터미널에서 `/mcp`로 승인해야 Notion 적재 단계 진행 가능. (정찰 단계는 승인과 무관하게 선행 가능)
- Notion 적재 위치: **새 최상위 페이지 생성**으로 확정.

## 8. 성공 기준

1. 세 사이트 각각에 대해 정찰 노트와 원본 샘플이 로컬에 존재한다.
2. Notion에 인덱스 페이지 + 카탈로그 DB가 생성되고, 각 소스별 데이터셋 행이 실데이터 스냅샷과 함께 채워진다.
3. 카탈로그의 각 행이 **재현 가능한 취득방법**(URL/API/파라미터)을 담아, 나중에 실제 데이터를 다시 가져올 수 있다.
4. "어느 사이트 → 어떤 데이터 → 어떻게 활용" 을 Notion에서 필터/질의로 즉시 찾을 수 있다.

## 9. 범위 밖 (Non-goals)

- 자동 갱신 파이프라인/스케줄러 구축 (수동 재현법만 기록).
- GIS/공간데이터 심화, 유료·회원 전용 데이터.
- 실제 제품/서비스 기획 자체(이번 산출물은 그 이전의 참조 레이어).
- 대량 크롤링/스크래핑 저장소 구축(대표 샘플 위주).

## 10. 위험 요소

- 빅데이터플랫폼의 익명 JSON API가 세션 토큰/Referer 등에 의존할 수 있어 재현법이 취약할 수 있음 → 캡처 시 필요한 헤더까지 기록.
- 사이트 구조 변경 시 카탈로그가 낡을 수 있음 → 인덱스 페이지에 "최종 정찰 일자" 명시.
- Notion MCP 미승인 시 적재 단계 전체가 대기 → 사용자 액션 의존.
